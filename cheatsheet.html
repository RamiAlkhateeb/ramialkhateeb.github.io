Prompting Cheatsheet

Claude, Gemini & Claude Code — copy, adapt, paste.

The four-part prompt formula

Every good prompt answers these four questions. Skip one and you get a vague answer.

Part Answers Example
Context What is this? Node/Express API, PostgreSQL, deployed on Railway
Material What do you have? Paste the real code or error — never describe it
Goal What does "done" look like? Return 400 with a field-level validation message
Constraints What must stay the same? Keep authMiddleware untouched
Template
I'm working on [context]. Here's the relevant code/error: [paste]. I want [goal]. Keep [constraint] unchanged.
Example
I'm working on a React + TypeScript dashboard using React Query. Here's the hook: [paste useOrders.ts]. I want it to retry failed requests up to 3 times with exponential backoff. Keep the return shape ({ data, isLoading, error }) unchanged.

Prompt templates, by moment

1Starting a feature

Prompt
I'm adding [feature] to [module]. Stack: [tech]. Existing code: [paste]. Before writing anything, give me a short plan — what files change, what's new, how it fits the existing pattern.

2Debugging

Prompt
I'm getting [error] in [file/function] when [trigger]. Code: [paste]. Stack trace: [paste]. I expected [behavior] instead. What's the cause and how do I fix it?

3Code review before a PR

Prompt
Review this [class/function] for: [issue 1], [issue 2], [issue 3]. Don't change [fixed thing] — that's intentional. [paste code]

4Test generation

Prompt
Write tests for this [function/method]. Cover: the normal case, [edge case 1], [edge case 2]. Our test style: [paste example]. [paste function]

5Understanding legacy code

Prompt
This is legacy code I inherited. I understand what it does, not why it's structured this way — especially [part]. [paste code]. What problem was this solving? Still necessary given [modern alternative]?

6Refactoring

Prompt
Refactor this to [goal]. Don't change external behavior or the public API. Code: [paste]. Style reference: [paste]

7Architecture discussion (no code yet)

Prompt
Deciding between [A] and [B] for [problem]. Context: [details]. What are the tradeoffs given our setup? Not asking for code yet — want to think this through first.

8Claude Code — plan before executing

Prompt
I want to [goal] in [file/area]. Use plan mode: give me a short plan — which files, how it fits [existing pattern], what could break. Don't write code yet.

9Claude Code — reviewing a diff

Prompt
Explain in plain language what this diff changes vs. before, and flag anything risky or unexpected. Diff: [paste]

10Batched fix request

Prompt
Fix these in [file]: (1) [issue 1] (2) [issue 2] (3) [issue 3]. Don't touch anything not listed. [paste code]

11Porting between frameworks

Prompt
Convert this [source] code to [target]. Keep the same logic and structure — this is a port, not a rewrite. Flag anything that doesn't translate directly. [paste code]

12Commit message / PR description from a diff

Prompt
Write a commit message and short PR description for this diff. Focus on why, not just what changed. [paste diff]

Before / after

Broken in production

Bad
My API is broken, can you help?

→ generic troubleshooting checklist.

Good
Checkout fails ~1 in 20 requests. Node/Express + Stripe. Error log: [paste]. Handler: [paste]. Started after [commit/date]. Likely cause?

→ targeted hypothesis.

Improving a component

Bad
Make this component better.

→ unpredictable, unwanted changes.

Good
Improve this component's re-render performance — it re-renders on every keystroke in a sibling input. Don't change props or styling. [paste]

→ fix scoped to the actual problem.

Good vs bad

Bad Good
"Fix this" "Fix this to return null instead of throwing when not found — don't change the signature"
"Why is this slow?" "This query takes 4s on 50k rows. Query + schema: [paste]. Where's the bottleneck?"
"Explain this code" "Explain why this is structured this way, especially the recursive call on line 40"
"Add a delete button and wire it up" "Plan how to add a delete button to UserRow.tsx — don't code yet"

Claude vs Gemini

Claude / Claude Code Gemini
Multi-step reasoning, architecture talks Quick factual lookups
Strict constraints followed Very large context (long logs, big docs)
Multi-file edits, planning before coding "Good enough" answers when Claude usage is low
Sustained context across a session Simple rewrites, translations, brainstorming

When not to use AI

  • Skip it — rename a variable, fix a typo, one-line config
  • Use Gemini / a smaller model — syntax lookups, simple formatting, early brainstorming
  • Use Claude / Claude Code — multi-step, multi-file, architecture, real debugging
Gut-check
Does this need planning or multi-file context — or am I just looking something up?

CLAUDE.md — what actually works

CLAUDE.md loads on every session — it's not free. Long, bloated files cause Claude to ignore your actual rules. Treat it like a scarce resource, not a wiki.

  • Keep it under ~200 lines. If it scrolls more than two screens, you're paying for it every message.
  • Lead with exact commands — build, test, lint, run. Highest ROI section by far.
  • Never duplicate your linter/formatter. Use a hook for that instead.
  • Give every rule a why. "Server components by default — client components caused 8s LCP regressions" survives; "write clean code" gets ignored.
  • Run /init to generate a starter, then cut it down. It's a first draft, not a finished file.
  • Put occasional/domain knowledge in Skills, not CLAUDE.md. Skills load on demand; CLAUDE.md loads every time.
  • Check it into git. Review it when Claude misbehaves, prune regularly.
Template
# Commands Build: [exact command] Test: [exact command] Lint: [exact command] # Architecture [stack, e.g. Next.js + Prisma + PostgreSQL] Domain logic lives in [folder structure]. # Rules (with reasons) - [Rule] — because [what broke before] - [Rule] — because [what broke before] # Don't - Introduce new dependencies without asking - [anti-pattern this project has been burned by]

Claude Code — session hygiene

  • Use Plan Mode before letting it edit. Skipping this is how you get a plausible-looking change that misses the point.
  • /clear between unrelated tasks. Old context costs you even when it's irrelevant now.
  • Went down the wrong path twice? Don't keep correcting — /clear and restart with a sharper prompt.
  • Use hooks for anything that must happen every time (formatting, lint-on-save, blocking writes to a sensitive folder). CLAUDE.md is advisory; hooks are deterministic.
  • Scope exploration. Point it at specific files or folders, not "go investigate the codebase."
  • Read the diff yourself before accepting.

Usage-saving habits

  • Front-load, don't follow up — one complete prompt beats five small ones
  • New topic = new chat//clear — old context costs tokens even when irrelevant
  • Batch related fixes into one prompt
  • Put recurring context in CLAUDE.md — don't retype it
  • Match model size to task size
  • Mind rolling usage windows — space out non-urgent tasks near a limit

The 10-second check before you hit send

Ask yourself
If a new developer joined this project with zero context, would this message alone be enough for them to help?

If no — add the missing context before sending, not after.
